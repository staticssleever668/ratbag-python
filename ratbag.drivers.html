probe(device)

Probe the device for information. On success, the driver will create a ratbag.Device with at least one profile and the appropriate number of buttons/leds/resolutions and emit the device-added signal for that device.

A caller should subscribe to the device-added signal before calling this function.

If device is a pathlib.Path object, the driver creates initializes for the device at that path. Otherwise, device is used as-is as backing device instance. This is used when emulating devices - note that the exact requirements on the device behavior may differ between drivers.

Completion of this function without an exception counts as success.

Parameters

  • device – The path to the device or a fully initialized instance representing a device

  • device_info – Static information about the device from external sources (system, data files, etc).

  • config – Driver-specific device configuration (e.g. quirks).

class ratbag.drivers.hidpp20.Hidpp20Driver

Bases: ratbag.drivers.Driver

Implementation of the Logitech HID++ 2.0 protocol.

Driver options supported:

  • Buttons: the number of buttons exported by the device

  • DeviceIndex: the HID++ device index

  • Leds: the number of LEDs exported by device

  • ReportRate: fixed report rate

  • Quirk: the quirk to apply to this device, one of G305, G602

Parameters

  • config – A dict of the (lowercase) driver options

  • quirk – None or one of Quirk

class ratbag.drivers.Rodent(path=None)

Bases: gi.overrides.GObject.Object

An class abstracting a physical device, connected via a non-blocking file descriptor. This class exists so we have a default interface for communicating with the device that we can hook into for logging and others.

See from_device() for the most convenient way to create a new Rodent object.

Parameters

path (Optional[Path]) – the path to the physical device

Note

The name Rodent was chosen to avoid confusion with ratbag.Device.

name

The device’s name (as advertized by the device itself)

path

The device’s path

report_descriptor

The bytes for this hidraw device’s report descriptor (if this device is a hidraw device, otherwise None)

GObject Signals:

  • data-to-device, data-from-device: the bytes that have been written to or read from the device.

  • ioctl-command: an ioctl has been called on the device with the given ioctl name and the data in bytes

  • ioctl-reply: an ioctl has returned data in bytes

These signals are primarily used by recorders, there should be no need to handle those signals in other parts of the implementation.

classmethod from_device(device)

A simplification for drivers. If the given device is already a pre-setup device (from a recorder), this function returns that device. Otherwise, this function returns a ratbag.drivers.Rodent instance for the given device path.

class ratbag.drivers.Driver

Bases: gi.overrides.GObject.Object

The parent class for all driver implementations. See ratbag/drivers/drivername.py for the implementation of each driver itself.

A driver must implement the DRIVER_LOAD_FUNC() function, it is the entry point for ratbag to instantiate the driver for the device.

DRIVER_LOAD_FUNC = 'load_driver'

The name of the function to instantiate a driver. A driver file must contain at least the following code:

class MyDriver(ratbag.Driver):
    def probe(self, device, info, config):
        pass

def load_driver(driver_name=""):
    return myDriver()

GObject Signals:

  • device-added: emitted for each ratbag.Device that was added during probe()
